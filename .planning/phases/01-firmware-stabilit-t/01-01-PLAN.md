MEM-01: Heap-Leak in JsonBufferPool::release() fixen und RAII-Guard einfuegen.

**1. release() Bug patchen (Zeilen 359-372):**

Ersetze die gesamte `release()` Methode:
```cpp
// Gibt einen Puffer frei
void release(char* buffer) {
    if (!buffer) return;
    if (xSemaphoreTake(mutex, 100 / portTICK_PERIOD_MS) == pdTRUE) {
        for (int i = 0; i < JSON_BUFFER_COUNT; i++) {
            if (buffer == buffers[i]) {
                inUse[i] = false;
                xSemaphoreGive(mutex);
                return;
            }
        }
        xSemaphoreGive(mutex);
    }
    // KRITISCHER FIX: delete[] IMMER wenn nicht im Pool —
    // unabhaengig ob Mutex erfolgreich war
    delete[] buffer;
}
```

Wesentliche Aenderungen:
- `if (!buffer) return;` als Null-Guard am Anfang
- `delete[] buffer;` steht AUSSERHALB des Mutex-Blocks — wird immer ausgefuehrt wenn Buffer nicht im Pool gefunden wurde
- Vorher: `delete[]` war innerhalb des Mutex-erfolg-Blocks — bei Mutex-Timeout wurde Heap-Buffer nie freigegeben

**2. JsonBufferGuard RAII-Klasse einfuegen (nach Zeile 373, nach dem schliessenden `};` von JsonBufferPool):**

```cpp
class JsonBufferGuard {
public:
    char* buf;
    JsonBufferGuard() : buf(jsonPool.acquire()) {}
    ~JsonBufferGuard() { if (buf) jsonPool.release(buf); }
    // Nicht kopierbar
    JsonBufferGuard(const JsonBufferGuard&) = delete;
    JsonBufferGuard& operator=(const JsonBufferGuard&) = delete;
};
```

Hinweis: Die Guard-Klasse referenziert `jsonPool` direkt (globale Variable, Zeile 375). Die Klasse wird nach der `jsonPool`-Deklaration platziert, damit der Compiler die Variable kennt. Falls `jsonPool` vor der Klasse deklariert ist (Zeile 375), muss die Guard-Klasse NACH Zeile 375 stehen.

Die Guard-Klasse wird in diesem Plan noch NICHT in bestehende Handler eingebaut — das ist ein optionaler Refactor fuer spaeter. Der release()-Fix allein behebt den Memory-Leak.
